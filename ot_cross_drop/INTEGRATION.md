# Integration into coineasy-forwarder

기존 `coineasy-forwarder`에 플러그인 모듈로 통합. 세션·Railway 서비스·환경변수 그대로 재사용.

## 통합 단계

### 1. 모듈 드롭

`coineasy-forwarder/` 안에 이 폴더를 `ot_cross_drop/`으로 복사:

```
coineasy-forwarder/
├── main.py                   # 기존
├── forwarders/               # 기존 포워딩 핸들러들 (예시)
├── ot_cross_drop/            # ← 추가
│   ├── __init__.py
│   ├── monitor.py
│   ├── approval.py
│   ├── sender.py
│   ├── metrics.py
│   ├── template.py
│   └── config.py
└── requirements.txt          # 기존 (이미 telethon 있음)
```

### 2. main.py 한 줄 추가

forwarder의 entrypoint에서 client 생성 직후:

```python
from telethon import TelegramClient
from ot_cross_drop import register as register_ot

client = TelegramClient(SESSION, API_ID, API_HASH)

# ... 기존 forwarder 핸들러들 등록 ...

register_ot(client)        # ← OT cross-drop 플러그인 등록

await client.start()
await client.run_until_disconnected()
```

`register_ot(client)`이 monitor + approval 핸들러 모두 attach. 다른 코드 변경 불필요.

### 3. .env 추가

forwarder의 기존 `.env`에 OT_* 키만 append:

```bash
# OT cross-drop module
OT_SOURCE_CHANNEL=@Origintrailkr
OT_TARGET_CHANNELS=@OriginTrail,@OriginTrailClub
OT_APPROVAL_CHAT_ID=
OT_ADMIN_USER_IDS=
OT_AUTO_APPROVE=false
OT_SEND_DELAY_SECONDS=2
```

Supabase는 forwarder가 이미 쓰고 있으면 `SUPABASE_URL` / `SUPABASE_SERVICE_KEY` 그대로 재사용.

### 4. Supabase 테이블 추가

```sql
create table ot_cross_drops (
  id bigint generated by default as identity primary key,
  kr_msg_id bigint not null unique,
  drop_link text not null,
  drop_msg text not null,
  sent_at timestamptz not null,
  targets jsonb not null,
  created_at timestamptz default now()
);

create table ot_cross_drop_metrics (
  id bigint generated by default as identity primary key,
  kr_msg_id bigint not null,
  interval_minutes int not null,
  snapshot jsonb not null,
  captured_at timestamptz not null,
  created_at timestamptz default now()
);

create index on ot_cross_drop_metrics (kr_msg_id);
```

### 5. 권한 점검

- `@OriginTrail` broadcast인지 확인 → broadcast면 Žiga에게 contributor 권한 요청
- `@OriginTrailClub`은 일반 그룹이라 송신자 계정 join만 되어 있으면 OK
- forwarder가 쓰는 송신자 계정이 위 두 채널에 들어가 있어야 함

### 6. Deploy

Railway forwarder 서비스 redeploy. 별도 서비스 안 만듦, 기존 worker가 OT 핸들러까지 함께 처리.

## Approval flow

승인 채팅(`OT_APPROVAL_CHAT_ID`)에 drop draft 도착 → 텍스트 명령으로 처리:

| 명령 | 동작 |
|---|---|
| `/otapprove <kr_msg_id>` | 글로벌 채널 발송 + metrics 트래킹 시작 |
| `/otreject <kr_msg_id>` | pending 큐에서 삭제, 발송 안 함 |
| `/otedit <kr_msg_id> <new text>` | drop_msg 교체 후 즉시 발송 |

`<kr_msg_id>`는 preview 메시지 첫 줄(`id: ...`)에 표시됨.

## Known limitations

**Userbot inline keyboard**: 현재 송신자 계정이 userbot이라 Telegram MTProto가 inline keyboard 버튼을 렌더링하지 않음 (bot account 전용 기능). 그래서 텍스트 명령(`/otapprove`, `/otreject`, `/otedit`)으로 처리함. 콜백 핸들러 코드(`on_button`)는 향후 bot 토큰 마이그레이션 시를 위해 유지.

**Future bot-account migration**: 별도 bot 토큰(@xxx_bot)을 만들어 동일 채널에 admin으로 넣으면 inline keyboard 동작 복구. 그때:
1. 새 bot 세션을 별도 `TelegramClient`로 띄우거나
2. 동일 forwarder에서 두 번째 client 인스턴스 추가

2가지 옵션이 있고, 결정은 사용 패턴 보고 추후. `on_button` 핸들러는 코드 수정 없이 동작.

## Safety

- `OT_AUTO_APPROVE=false` 유지 (글로벌 노출 메시지, 톤 1~2주 검증 후 자동화 검토)
- 핸들러 prefix `ot` 사용 → forwarder의 기존 CallbackQuery와 충돌 없음
- pending 큐는 in-memory dict → 재배포 시 휘발. 보존 필요하면 Supabase row로 마이그레이션
